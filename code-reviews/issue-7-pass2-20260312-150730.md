## Go Persistence Layer & REST API Review

I've reviewed the Go persistence layer (`store` package), the contract tests, and the REST API surface from the provided code. My findings are categorized by **API Correctness**, **Edge Cases**, **Data Correctness**, and **Behavioral Correctness**. Each is rated **LOW**, **MEDIUM**, **HIGH**, or **CRITICAL** severity.

---

### 1. API Correctness

#### 1.1. Status codes & Error reporting
- **Finding:** All endpoints use standard status codes (200, 201, 202, 400, 404, 409, 422, 501, etc.). `writeError` provides structured error responses.
- **Severity:** PASS

#### 1.2. Consistent Response Shapes
- **Finding:** JSON return shapes conform to what clients would expect (e.g., object-wrapped arrays like `{"runs":[...]}`), with consistent snake_case keys. For `/v1/runs/{id}`, the field set matches the `store.Run` JSON structure.
- **Severity:** PASS

#### 1.3. Fields Present
- **Finding:** Some fields are intentionally omitted (e.g., `UsageTotalsJSON`, `CostTotalsJSON` are not surfaced in the API). This is justifiable, as they are marked with `json:"-"` in the store model, signaling API consumers shouldn't expect them.
- **Severity:** PASS

#### 1.4. Allow Header on 405
- **Finding:** Uses `Allow` header on 405. Confirmed.
- **Severity:** PASS

---

### 2. Edge Cases

#### 2.1. Empty/Whitespace IDs
- **Finding:** 
    - Create/Update in store impls: Both check `run.ID == ""` and error cleanly.  
    - API: If routes like `/v1/runs/` or similarly have `runID == ""`, these are 404'd or 400'd.
- **Severity:** PASS

#### 2.2. Filtering by Empty/Whitespace/Nonexistent Values
- **Finding:** 
    - List endpoints trim whitespace for filters.
    - Returns empty arrays if no results, not null or errors.
- **Severity:** PASS

#### 2.3. Concurrency
- **Finding:** 
    - SQLite: restricts to a single writer via `SetMaxOpenConns(1)` and uses WAL; contract tests for concurrent access pass.
    - Memory: explicit `sync.RWMutex`.
    - Contract tests for concurrent creation, updates, and event appending pass.
- **Severity:** PASS

---

### 3. Data Correctness

#### 3.1. All User Data Is Roundtrippable
- **Finding:** All user-persisted fields in create/update are persisted and retrieved. The contract test checks this.
- **Severity:** PASS

#### 3.2. Filters Are Actually Applied
- **Finding:** `ListRuns` in both store impls and handler applies filters only if non-empty; test covers all permutations (`Status`, `TenantID`, `ConversationID`).
- **Severity:** PASS

#### 3.3. Ordering
- **Finding:** All runs returned ordered by `created_at DESC`; messages and events ordered by `seq ASC` (contract tests enforce this).
- **Severity:** PASS

#### 3.4. Uniqueness & Duplicates
- **Finding:** Duplicate run IDs are rejected at store level (both impls return error); tested.
- **Severity:** PASS

---

### 4. Behavioral Correctness

#### 4.1. Store Not Configured (runStore == nil)
- **Finding:** All endpoints using runStore return `501 Not Implemented` (or `404` if not found in both in-memory and store), as expected. This is handled gracefully and tested.
- **Severity:** PASS

#### 4.2. Graceful Fallback: In-Memory → Store
- **Finding:** For `/v1/runs/{id}`, if not found in-memory, falls back to store, then 404 on both-miss. This is the expected and robust pattern.
- **Severity:** PASS

---

### 5. Other Notes

- Usage/cost totals fields are not exposed—which seems intentional (no obvious contract or test says otherwise).
- Test and API surface seem mature and well-covered for common and edge behaviors.
- SQLite uses timestamps as `TEXT`, which is adequately handled, and time zones are UTC-normalized.
