# Go Persistence Layer Correctness Review

This review addresses the following areas and rates each finding as **CRITICAL**, **HIGH**, **MEDIUM**, or **LOW**.  
The analysis covers interface/implementation contract correctness, transaction correctness, concurrent correctness, test coverage gaps, and schema correctness.

---

## 1. Interface/Implementation Contract Correctness

**a. MemoryStore implements the Store interface.**

- **Finding:** MemoryStore and SQLiteStore both conform to the Store interface contracts, and a test suite runs all contract tests against both.
- **Examination:** Both implement the semantics (e.g. duplicate CreateRun returns error, GetRun returns NotFound, ListRuns returns [] if empty).
- **Finding:** UpdateRun in MemoryStore does not check for run existence before update, whereas SQLiteStore relies on UPDATE with no rows affected—which may be acceptable if the contract is "upsert," but the doc says "The run must already exist."
    - In MemoryStore: `m.runs[run.ID] = cp`
    - In SQLiteStore: no explicit check, so updates rows=0 is silent.
- **Severity:** **MEDIUM**
    - **Details:** The contract says "must already exist", but both impls allow `UpdateRun` to succeed and create the row if missing. (SQLite: row not present = affected rows=0, but no error returned; MemoryStore: new row always created.)

**b. Enforced constraints on messages and events (seq must be unique per run).**
- **Finding:** SQLiteStore enforces seq (UNIQUE(run_id, seq)) in both run_messages and run_events. Insert duplicate: returns an error.
- **MemoryStore:** No check. Appends always.
    - If client calls AppendMessage/ApppendEvent with duplicate seq, MemoryStore happily grows the slice; SQLiteStore errors.
- **Severity:** **HIGH**
    - **Details:** MemoryStore behaves differently from SQLiteStore regarding duplicate message/event `seq` values. SQLiteStore will error/constraint-fail; MemoryStore will allow arbitrarily many at the same seq. This is user-visible if using MemoryStore and then migrate to SQLite, or for prod/test parity.

## 2. Transaction Correctness

- **Finding:** All multi-step operations (e.g. AppendMessage, AppendEvent, CreateRun, UpdateRun) are atomic in SQLiteStore due to single SQL statement per call.
    - In SQLite, each individual Exec is wrapped in a transaction by default (unless manually batched).
    - **No multi-object/multi-table operations** (e.g. see no pattern where CreateRun + AppendMessage must be in one tx).
- **Finding:** MemoryStore uses lock per operation, no transaction API.
- **Severity:** **PASS (no problems found)**

---

## 3. Concurrent Correctness

- **Both implementations are documented as thread-safe.**
- **SQLiteStore:** Relies on sql.DB's concurrency and sets SetMaxOpenConns(1) to serialize writers, as SQLite "single writer" rules advise. db is not closed until Close.
- **MemoryStore:** Uses fine-grained sync.RWMutex around every public operation. Deep-copies are used for return values to avoid aliasing across callers.
    - **Finding:** There are no apparent Go data races or logic races in MemoryStore.
- **PASS (no problems found)**

---

## 4. Test Coverage Gaps

Tests are thorough but there are small holes:

**a. AppendMessage/ApppendEvent with duplicate seq:**
- There is no explicit test for the behavior.
    - What if AppendEvent for (run_id, seq) already exists?
        - SQLiteStore will error, MemoryStore will happily append a duplicate.
    - Contract tests only check for one event/message of each, but not error on duplicate append.
- **Severity:** **HIGH**
    - **Details:** Without a test, MemoryStore will never enforce the SQL constraint parity. This could lead to missing test coverage of user-reported "duplicate event" problems if migrating from MemoryStore to SQLiteStore in prod.

**b. UpdateRun NotFound behavior:**
- The interface contract states that "The run must already exist (use CreateRun for initial persistence)". Neither implementation enforces this fully; neither does the test.
    - UpdateRun with unknown run ID results in "silent add" in MemoryStore, "updates 0 rows" in SQLiteStore (but does not check affected rows).
- **Severity:** **MEDIUM**
    - **Details:** The contract is not tested or enforced, and users may expect an error if updating a nonexistent run.

**c. Messages/Events for non-existent runs:**
- No test for appending a message/event to a non-existent run ID.
    - SQLiteStore will fail on FK, MemoryStore will happily insert.
- **Severity:** **HIGH**
    - **Details:** MemoryStore violates the FK constraint; prod/test parity is broken.

**d. Test coverage for large volumes/boundaries.**
- There are concurrent tests, but no explicit `ListRuns`/`GetEvents` with more than e.g. 1000 entries ("too many" case), or for max int/overflow.
- **Severity:** **LOW**
    - **Details:** Not likely critical unless store is expected to handle very large loads.

## 5. Schema Correctness

- **All fields mapped with not null/defaults.**
- **UsageTotalsJSON, CostTotalsJSON:** both NOT NULL DEFAULT ''. Safe even if never filled.
- **run_messages, run_events:** FK ON DELETE CASCADE. Duplicates controlled by UNIQUE(run_id, seq).
- **Finding:** The schema constraints (NOT NULL, UNIQUE, FK) match code expectations except for the following:
    - **Append to non-existent run:**
        - Schema will fail FK, MemoryStore will not—see above.
- **Severity:** **High (for test/prod parity, already mentioned above)**

# Conclusion

**FAIL**

## CRITICAL/HIGH Issues Identified

- **High:** MemoryStore does not enforce message/event seq uniqueness per run—test coverage does not require this.
- **High:** MemoryStore allows messages/events for missing runs (no FK); SQLite rejects them due to FK constraints—test coverage does not require this.
- **High:** Tests do not check for the above behaviors, so deviations persist unnoticed.

---

## **What to Fix**

**Address at least these:**
1. Add contract tests that check:
    - AppendMessage and AppendEvent with duplicate seq (should return error).
    - AppendMessage and AppendEvent to non-existent run (should return error).
2. Update MemoryStore to honor these constraints (reject as error on append duplicate seq, or on unknown run).
3. Optionally, clarify (fix) UpdateRun behavior: error on missing, and/or document upsert as accepted.
