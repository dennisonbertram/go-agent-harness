### 3) Concrete findings
- Built-in profiles are exactly six embedded TOMLs in `internal/profiles/builtins`: `github`, `file-writer`, `researcher`, `bash-runner`, `reviewer`, `full`.
- `full.toml` is currently:
  - default profile name for many docs
  - tool allow-list empty (`[]`) meaning all tools
  - `max_steps = 30`, `max_cost_usd = 2.0`, model `gpt-4.1-mini`, no system prompt.
- There is one canonical scripted smoke entrypoint already: `scripts/smoke-test.sh`, documented in `docs/runbooks/golden-path-deployment.md` as a manual smoke flow using `HARNESS_PROFILE=full` and `HARNESS_AUTH_DISABLED=true`.
- Existing helper scripts (`scripts/curl-run.sh`, `scripts/test-multiturn.sh`) provide runnable API-client flows but are not a single golden-path golden profile smoke suite.
- Integration test coverage already exists broadly for individual endpoints and behaviors:
  - Profile resolution and built-in loading in `internal/profiles/profile_test.go`
  - `run_agent` default-to-`full` behavior in `internal/harness/tools/deferred/run_agent_test.go`
  - Many `internal/server/http_*_test.go` tests cover health, providers/models, run lifecycle, continuation/steer/context/compact/todos/conversation, etc.
- A minimal safe golden-path suite for `full.toml` should be:
  - Start `harnessd --profile full` with ephemeral port + auth disabled for local smoke.
  - Verify `/healthz` (status 200), `/v1/providers` non-empty, `/v1/models` non-empty.
  - Create run `POST /v1/runs` with fixed prompt and poll `/v1/runs/{id}` until terminal.
  - Assert completed and exact output token (e.g., `SMOKE_TEST_PASS`).
  - Read `/v1/runs/{id}/events` and confirm stream replay contains terminal evidence (`run.completed` or equivalent event flow).
  - Optional + minimal persistence check (only when `HARNESS_CONVERSATION_DB` set): verify conversation/threading via a continue or conversation endpoint.
  - Optional one-tool check: run deterministic no-op `bash` prompt and verify run completes and output contains expected marker.
- There is no existing end-to-end scripted/automated "golden path around `full.toml`" in one place; issue-361 grooming currently labels this as still missing.

### 4) Risks/blockers
- Issue dependency note remains: issue #361 is marked blocked by issue #362 (provider bootstrap/startup semantics).
- Conversation persistence tests require `HARNESS_CONVERSATION_DB`; without it they are expected to 501 and should be treated as conditional, not failures.
- Provider-dependent behavior can flap (quota/model compatibility/pricing), so suite should avoid strict assertions on cost accounting and keep provider-specific assertions minimal.
- Minor docs drift: an older issue-237 note describes different `full` defaults (50 steps, different costs) vs current `full.toml` (30 steps), so docs should be aligned before formalizing expected thresholds.
