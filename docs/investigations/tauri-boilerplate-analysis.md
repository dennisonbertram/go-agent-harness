## 4. Communication Patterns

### Client-Server Architecture
- **HTTP Base:** http://localhost:{DYNAMIC_PORT} (set by init.sh)
- **No hardcoded ports:** Defaults to 3131 (server) & 1420 (Vite), but assigned dynamically to support worktrees
- **CORS Enabled:** Tauri loopback origin + Vite dev server origin

### Streaming & SSE
The app uses **Vercel's `ai` SDK** with **DefaultChatTransport**:
1. **useChat Hook** (`@ai-sdk/react`): React hook managing message state, streaming
2. **POST /api/chat**: Main endpoint for starting conversations
3. **Streaming Response:** AI SDK handles `ReadableStream` from server
4. **Event Processing:** Client processes streaming events (`content`, `tool-calls`, `finish-reason`, etc.)

**Key Stream Events Handled:**
- `message_start` / `message_end` — Chat boundaries
- `content_block_start` / `content_block_delta` / `content_block_end` — Text chunks & tool calls
- `tool_use` — Tool invocation with arguments
- `thinking` — Claude's thinking blocks
- `usage` — Token counts (cumulative)
- `cache_creation_input_tokens`, `cache_read_input_tokens` — Prompt cache metrics

**No WebSocket:** Uses traditional HTTP long-polling via fetch (AI SDK handles this).

### Custom Protocol Integration
- **Tauri IPC:** Desktop app communicates with Rust shell via `@tauri-apps/api`
- **Sidecar Binary:** Server can run as embedded Tauri sidecar (binary bundled in app)
- **Bearer Token Auth:** SIDECAR_BEARER_TOKEN enforced in production (skipped in dev)

### API Routes (Backend — Hono)
```typescript
GET  /api/health
POST /api/auth/...
POST /api/chat                    // Start conversation
GET  /api/chat/...                // Retrieve chat history
POST /api/sessions                // Create/manage sessions
GET  /api/workspaces/...          // Workspace operations
POST /api/mcp                      // MCP server management
GET  /api/git/...                 // Git operations
POST /api/teams/...               // GitHub integration
POST /api/linear/...              // Linear integration
POST /api/plaid/...               // Plaid bank linking
POST /api/google/...              // Google OAuth flows
... (40+ route groups total)
```

## 6. Build & Run Instructions

### Prerequisites
```bash
# Required
pnpm install -g pnpm              # Package manager
brew install bun                  # Server runtime (macOS)
rustup install stable             # Rust for Tauri

# Required: Claude subscription (not API key)
# Do NOT set ANTHROPIC_API_KEY in dev — use subscription auth
```

### Setup (One-Time)
```bash
cd claude-tauri-boilerplate
./init.sh                         # Checks deps, installs packages, starts dev servers
```

### Running (Development)

**Recommended (Web mode with dynamic ports):**
```bash
./init.sh
# Creates .init-state with $SERVER_URL and $FRONTEND_URL
# Prints health check URL
```

**Worktree mode (background + isolated ports):**
```bash
INIT_DAEMONIZE=1 ./init.sh
source .init-state
echo $SERVER_URL      # e.g., http://localhost:3847
echo $FRONTEND_URL    # e.g., http://localhost:1623
```

**Stop background services:**
```bash
./init.sh stop
```

**Desktop app (native Tauri window):**
```bash
pnpm --filter @claude-tauri/desktop tauri dev
# Requires Rust + Xcode Command Line Tools
```

### Building

**Web bundle:**
```bash
pnpm build
# Output: apps/desktop/dist/ (Vite) + apps/server/dist/ (Bun)
```

**Tauri desktop app:**
```bash
pnpm --filter @claude-tauri/desktop tauri build
# Output: .dmg, .app, or .exe depending on platform
```

**Sidecar binary (embeds server):**
```bash
pnpm build:sidecar
# Compiles Bun server to standalone binary for bundling with Tauri app
```

### Testing
```bash
pnpm test                         # All tests (desktop + server)
pnpm --filter @claude-tauri/desktop test -- --run
cd apps/server && bun test        # Backend only
```

### Key Environment Variables

- `useStreamEvents` hook — Extracts, parses, and normalizes stream events
- `DefaultChatTransport` integration — Handles HTTP transport layer
- Event mapper (`event-mapper.ts` in server) — Converts Claude SDK events to UI domain

**Backend Routes (✓ Can Be Stripped or Repurposed)**
- `/api/chat` endpoint — Core conversation handler, reusable as-is
- `/api/sessions` — Session persistence logic (SQLite queries)
- `/api/mcp` — MCP server discovery and tool listing
- Error handler middleware — Standardized JSON error responses

### What Requires Significant Rework (Heavy Dependencies)

**Workspace/Git Integration (✗ High Coupling)**
- Tightly integrated with git worktree management
- Assumes local project repos with clone/branch logic
- **Option:** Strip workspace views, keep chat core

**External Integrations (✗ Must Be Removed/Replaced)**
- GitHub, Google, Linear, Plaid, Discord, Twitter connectors
- ~37 connector modules in `/connectors/`
- **Option:** Delete `/routes/teams`, `/routes/linear`, `/routes/plaid`, etc.; use only core chat routes

**Subagent System (✗ Complex Orchestration)**
- Multi-agent task queue with specialized worker
- Depends on persistent storage (sessions, checkpoints)
- **Option:** Use as-is (powerful) or simplify to single-agent

**MCP Server Management (○ Moderate)**
- Resolves MCP configs, validates tool schemas
- **Option:** Keep—it's how Claude tools are loaded; or use minimal subset

### CSP (Content Security Policy)
- Allows `ipc:` for Tauri communication
- Allows `http://localhost:3131` for server
- Allows `http://ipc.localhost` for sandbox isolation
- Restricts external domains (can be extended for APIs)

### Deep Linking
- **Scheme:** `claudetauri://`
- **Use case:** Open specific sessions, workspaces, or documents from external tools

### Database
- **Location:** `~/.claude-tauri/data.db` (shared across all clones by default)
- **Option:** Use `DB_DIR=./data` or `DB_PATH=./data/data.db` for project-local DB
- **Embedded:** Uses `bun:sqlite` (no separate server needed)
- **Schema:** Migrations in `/apps/server/src/db/` (CREATE TABLE statements)

### Auth Flow
1. **Desktop app launched** → Checks `SIDECAR_BEARER_TOKEN` env var
2. **Dev mode:** No token required (auth skipped)
3. **Production:** Token required for API access (enforced via middleware)
4. **Claude subscription:** SDK automatically prompts for Claude CLI auth (like `claude` CLI)

## 9. Key Architectural Patterns

### Monorepo Structure
- **pnpm workspaces:** Three independent packages with shared types
- **Apps:** `desktop` (frontend) + `server` (backend)
- **Packages:** `shared` (types) + `pdf-forms` (utility)
- **Fast installs:** pnpm's hard-link store (~2-3s with warm cache)

### API Design
- **Hono framework:** Lightweight, TypeScript-first HTTP server
- **Route grouping:** Organized by domain (chat, sessions, git, teams, etc.)
- **Error handling:** Centralized middleware returning `{ error, code, details? }`
- **Bearer auth:** Optional token validation (production-only)

### State Management
- **Frontend:** React hooks (useChat, useStreamEvents, useCostTracking, etc.)
- **Backend:** SQLite queries + in-memory maps for permission/plan state
- **Persistence:** Sessions, messages, checkpoints stored in SQLite

### Streaming Architecture
1. **Client:** `useChat` hook sends POST to `/api/chat`
2. **Server:** Builds `StreamingContext`, calls `buildStreamExecute()`
3. **Claude SDK:** Iterates over stream, yields events
4. **Response:** Wraps events in AI SDK protocol, sends via ReadableStream
5. **Client:** `useStreamEvents` hook extracts tool-calls, thinking, usage

### Testing Strategy
- **Frontend:** Vitest + React Testing Library (colocated `.test.tsx` files)
- **Backend:** Bun test runner (colocated `.test.ts` files)
- **TDD requirement:** Tests must exist before code is merged (mandatory in CLAUDE.md)

## 10. Notable Technical Decisions

1. **No WebSocket:** Uses HTTP streaming via `ai` SDK's `DefaultChatTransport` (simpler, works over HTTP)
2. **No Redux/Zustand:** Pure React hooks for state (less boilerplate, easier to follow)
3. **Bun over Node.js:** Faster startup, built-in SQLite, single binary (sidecar optimization)
4. **Tailwind v4:** Just-in-time compilation, excellent DX, no CSS module overhead
5. **Monorepo isolation:** Shared types prevent client-server mismatch
6. **Dynamic ports:** Enables parallel worktrees without collisions
7. **SQLite as primary storage:** No external database, lower operational overhead
8. **MCP server abstraction:** Tools are resolved at runtime, enabling plugin-style extensibility
9. **Markdown-first chat:** Renderers registered in a map, allowing custom UI per tool type
